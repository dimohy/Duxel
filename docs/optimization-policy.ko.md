# 성능 최적화 기록 정책

> 마지막 동기화: 2026-07-10

이 문서는 Duxel에서 성능/메모리 최적화를 수행할 때 **반드시 남겨야 하는 기록 기준**을 정의합니다.

## 1) 기록 원칙

- 최적화 1건마다 반드시 로그 1건을 남긴다.
- 로그는 "가설 → 변경 → 검증" 순서로 기록한다.
- 수치 검증은 가능한 한 동일 샘플/동일 옵션으로 A/B 비교한다.
- 결과가 음수(성능 하락)여도 숨기지 않고 그대로 기록한다.
- 저장소 벤치는 `global.json`에 고정된 SDK(`10.0.301`, patch roll-forward만 허용)로 실행하고, 로그에 `dotnet --version`을 남긴다.

## 2) 필수 로그 항목

각 항목을 누락 없이 기록한다.

- 날짜/세션 식별자
- 변경 영역 (파일/모듈)
- 최적화 유형 (GC, 풀링/재사용, 비동기화, 핫패스 리팩토링 등)
- 변경 요약 (1~3줄)
- 검증 명령어
- Before / After 수치
- 개선율(%)
- 리스크/후속 과제
- SDK 버전, 워밍업 조건, 독립 실행 횟수
- 평균 FPS와 raw frame 기준 median/p95/p99 frame time, 1% low FPS

## 3) 로그 저장 위치

- 정책 문서: `docs/optimization-policy.ko.md`
- 세션 로그: `docs/optimization-session-YYYY-MM-DD.md`

## 4) 검증 기준

- 최소 1회 `dotnet build Duxel.slnx -c Release`
- 최소 1회 런타임 벤치 실행
  - 벡터: `./scripts/run-vector-clip-ab.ps1 ...`
  - 레이어 혼합: `./scripts/run-layer-widget-clip-ab.ps1 ...`
- 각 후보는 phase 측정 전에 최소 `0.25`초 또는 `96`프레임을 워밍업하고, 같은 조건으로 최소 3회 독립 실행한다.
- 비교값은 독립 실행 3회 이상의 중앙값을 사용한다. 평균 FPS만으로 결론 내리지 않고 raw frame sample에서 계산한 median/p95/p99 frame time과 가장 느린 1% frame의 조화 평균인 1% low FPS를 함께 기록한다.
- 순서 영향을 받을 수 있는 A/B phase는 실행마다 순서를 교대하거나 별도 프로세스로 분리하고, 결과 artifact에 실제 phase 순서를 기록한다. 자동 판정은 3회 이상 반복 중앙값과 noise 구간을 함께 사용한다.
- 벤치 결과는 phase별 표, 각 지표의 Before/After, 개선율(%)을 함께 기록한다.
- A-B-A-B 재검증에서 개선 방향이 뒤집히거나 실행 간 변동 구간이 겹쳐 결론이 달라지면 해당 정량 결과를 `INVALID`로 표시한다. 이때 성능 개선율을 주장하지 않는다. 소비처가 없다는 것이 코드 경로로 증명된 dead work 제거 같은 구조 변경은 정성 결과로 유지할 수 있지만, 정량 개선으로 합산하지 않는다.

## 5) 표준 개선표 형식

| 항목 | 유형 | 대상 | 검증 | 결과 |
|---|---|---|---|---|
| 예시 | GC | `UiDrawCommand` | vector A/B | +1.2% |

## 6) 운영 규칙

- 새 최적화 PR/커밋에는 해당 세션 로그 링크를 포함한다.
- 동일 영역 재최적화 시 기존 로그를 덮지 않고 새 행을 추가한다.
- 측정이 어려운 구조 리팩토링은 정성 결과를 쓰되, 가능한 가장 근접한 지표를 함께 첨부한다.
