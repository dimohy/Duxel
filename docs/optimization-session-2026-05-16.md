# Optimization Session — 2026-05-16

## Scope

CPU-side reduction for immediate-mode frames, with the goal of feeding the Vulkan renderer more consistently instead of repeatedly rebuilding or copying geometry on the CPU.

Important continuation note:

- The optimization target is **not limited to static-cache performance**. Static cache is only one smaller part of the overall optimization direction.
- Dynamic rendering performance is also a first-class target: dynamic primitive generation, dynamic upload, command recording, pipeline/descriptor churn, and CPU-side frame construction must continue to be optimized.
- Do not accept a static-cache FPS gain if dynamic-heavy phases or visible correctness regress. Benchmark validation must include both static-heavy and dynamic-heavy samples.
- Earlier Layer Canvas black/blank body regressions have been fixed by retaining primitive arrays in static layer replay. Future scroll/clip/static-cache changes must still include screenshot validation for off-screen-then-visible layer bodies.
- Static cached layer capture must not be clipped to the first visible parent viewport. Build static layer contents with a full local capture clip, then intersect command clips with the current parent local clip during replay.

- Windows workspace
- Managed FBA benchmark path through `run-fba.ps1`

## Interpretation

The major CPU waste was not Vulkan upload after the static cache was warm; it was the core-side static reference append path copying the cached static vertex/index/command buffers every frame. Moving retained caches to `AppendRetainedStaticReference()` drops the warmed static-cache UI render slice from hundreds of microseconds toward roughly one tenth of a millisecond on this run, and raises static-cache FPS by about forty percent.

The renderer still spends most steady-state time in submit/present plus normal command recording. GPU utilization will not naturally reach 90% in this benchmark unless the workload becomes GPU-heavy enough. This optimization lowers CPU pressure so the next meaningful GPU-utilization step is architectural: either retained GPU layer composition or GPU-instanced primitive rendering.

The dynamic primitive improvement confirms that CPU-side shape tessellation is still a key dynamic-work bottleneck. Precomputing common circle unit vertices is a low-risk step; the structural endpoint remains GPU-instanced primitive rendering so dynamic circles/rectangles can be submitted as compact instance data rather than expanded CPU-side triangles.

The first GPU-instanced circle implementation validates that removing CPU tessellation is highly beneficial for dynamic-heavy workloads. The initial implementation caused a visual regression because the circle pipeline reused vertex binding `0`, which could invalidate regular geometry/text binding state. Moving circle instances to binding `1` fixed the visual issue in captured runs. The specialized circle command append path then recovered static/mixed-cache performance while preserving the compact dynamic primitive path.

The first GPU-instanced filled-rect implementation confirms that removing CPU-side quad expansion can help rect-heavy dynamic/layer workloads, especially the all-dirty layer and mixed nocache phases. The short global benchmark regressed, so the next optimization should focus on command recording state churn: primitive pipeline switches, descriptor binds to the white texture, and opportunities to batch compatible rect commands without disturbing text/triangle ordering.

Caching the primitive instance-buffer binding removes redundant `vkCmdBindVertexBuffers` calls on contiguous primitive commands and improves dynamic-heavy benchmark phases. Static-heavy phases are still sensitive to command ordering and descriptor/pipeline transitions, so the next structural target is reducing white-texture descriptor binds and primitive pipeline switches while preserving visual order.

The Layer Canvas visual regression came from the static layer cache capture path, not Vulkan drawing itself. Once filled rectangles and circles became primitive buffers, cached layer replay still retained only vertices/indices/commands. Static layer bodies therefore replayed commands that referenced missing primitive arrays. Capturing and replaying primitive arrays restores the intended layer contents.

Re-enabling static secondary command buffers for primitive-bearing static draw lists recovers mixed cached performance while preserving Layer Canvas correctness. The cache key must include primitive instance buffers because the recorded secondary command buffer binds those device-local buffers directly.

The white texture resource lookup fast path was tested and rejected. It preserved visual correctness but added extra branch/cache state on every texture lookup and did not reduce the dominant static/mixed costs in measured runs. The retained codebase therefore stays on the simpler dictionary lookup path, and future work should focus on reducing primitive pipeline transitions or increasing secondary command reuse.

Deferring regular geometry buffer binding is a lower-level state reduction that preserves draw order and removes redundant binding work from primitive-only command streams. The longer repeat improved global and most cached/mixed phases, but high-density nocache mixed remained worse. That points to remaining command-order and primitive pipeline switching costs rather than geometry binding alone.

Moving scissor rejection before texture lookup was also rejected. It can help clipped dynamic command streams, but it regressed cached-heavy phases sharply in the measured mixed cache-170 workload. The next retained optimization should therefore be measured at the command ordering or pipeline switching level, not another isolated per-command branch shuffle.

The command-signature reuse pass separates static geometry buffer lifetime from command metadata changes. This is intended for scroll/clip/translation cases where static layer geometry is unchanged but command clip rectangles or push translations vary. Fixed-position A/B benchmarks were mixed: layer all-dirty and mixed density-100 improved, while global and high-density cache phases regressed in short managed runs. The dedicated scrolling static-layer benchmark then exposed two scroll-path issues. First, layer replay included translation in the static geometry key, so every scroll position looked like a new static GPU buffer. Removing translation from that key turns the measured static-scroll path from multi-millisecond uploads into low-microsecond checks while keeping command signatures conservative for clip/translation changes. Second, the cache build itself was still clipped to the first visible parent viewport; that could cache off-screen local regions as empty. Static layer build now uses a full local capture clip and leaves viewport clipping to replay command metadata.
